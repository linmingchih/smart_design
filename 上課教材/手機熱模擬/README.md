# 手機熱模擬技術導讀

本專案使用 PyAEDT `0.26.3` 作為自動化框架，透過 `Icepak` 將手機 3D CAD 幾何匯入 Ansys Electronics Desktop，建立材料、熱源、網格與求解流程，最後讀回溫度監測結果。整體目標不是手動在圖形介面中逐步設定，而是把一套可重複執行的手機熱模擬流程腳本化。

## 這個腳本在做什麼

從高層來看，`main.py` 串起了一個完整的 Icepak 前處理到後處理流程：

1. 啟動 Icepak 工作階段，並指定 AEDT 版本。
2. 匯入手機外殼與內部零件的 STEP 幾何。
3. 建立或補充材料資料，例如塑膠類材料。
4. 依據零件名稱規則，自動指派體材料與表面材料。
5. 對不同材料群組設定求解優先順序，降低幾何重疊時的判定衝突。
6. 對指定封裝元件加入功耗，模擬晶片或封裝發熱。
7. 建立求解設定、外部空間與網格設定。
8. 執行熱分析，並讀取溫度監測點結果。

這表示它比較像是一個「自動化分析流程樣板」，而不是單純匯入模型後立即求解的最小示範。

## 技術架構概觀

本流程的核心技術組成如下：

- PyAEDT `0.26.3`：Python 與 AEDT 的自動化介面。
- Icepak：Ansys 用於電子散熱與熱流分析的求解器。
- STEP CAD：作為手機結構幾何的輸入來源。
- 名稱規則驅動的材料指派：利用物件名稱關鍵字，快速把材料套到大量零件上。
- 功耗驅動的熱源建模：將封裝元件視為主要熱源。
- 監測點回讀：在求解後自動擷取重點元件溫度。

對教學或原型驗證而言，這種做法的優點是流程一致、可重跑、可修改，也容易逐步擴充成更完整的熱設計自動化腳本。

## 典型資料流

可以把整個執行流程理解成下面這條資料流：

```mermaid
flowchart LR
    A[STEP 幾何檔] --> B[PyAEDT 啟動 Icepak]
    B --> C[材料與表面規則指派]
    C --> D[封裝功耗設定]
    D --> E[區域與網格建立]
    E --> F[Icepak 求解]
    F --> G[溫度監測結果輸出]
```

這裡最重要的觀念是，腳本不是逐一手動選取零件，而是依靠「幾何名稱」、「材料規則」與「功耗對應表」來自動建模。只要 CAD 命名邏輯穩定，整套流程就能反覆套用到相似機種或不同設計版本。

## 主要輸入與輸出

### 輸入

- AEDT / Icepak 執行環境。
- PyAEDT `0.26.3`。
- 手機 3D CAD 檔案，例如 STEP。
- 材料對應規則。
- 封裝功耗清單。

### 輸出

- 已完成材料與熱源設定的 Icepak 模型。
- 完成求解後的溫度結果。
- 監測點對應的溫度數值字典，方便後續整理或匯出。

## 腳本設計重點

這份腳本反映了幾個常見的電子熱模擬自動化思路：

- 以命名規則取代大量手動點選。
- 將材料、功耗與網格設定集中在腳本內，方便版本管理。
- 把前處理、求解、結果讀取整合成一次執行。
- 用監測點鎖定重點封裝，而不是一開始就追求完整後處理報表。

這樣的設計特別適合課程示範、流程標準化，以及早期架構評估。當模型成熟後，還可以再往下延伸到更細的接觸熱阻、自然對流 / 強制對流條件、更多材料資料，或更完整的結果輸出格式。

## 使用時要注意的事情

雖然這份腳本已經具備自動化骨架，但實際使用時仍有幾個前提：

- CAD 物件名稱必須和材料規則一致，否則零件可能被略過。
- 功耗表中的封裝名稱必須存在於模型內，才能成功加熱源。
- AEDT 版本要和執行環境相符。
- 材料資料與網格參數是否合理，仍需依手機結構與分析目的調整。

換句話說，這份腳本比較像是「可工作的分析流程起點」，真正要拿來做設計決策時，還需要搭配實際機構命名規範、材料庫與驗證流程。

## 適合的應用情境

這個範例特別適合以下情境：

- 手機或消費性電子產品的熱模擬教學。
- 建立 CAD 匯入到熱求解的標準作業流程。
- 快速驗證不同封裝功耗配置對溫度分布的影響。
- 作為更完整 PyAEDT / Icepak 自動化專案的起始版本。

## 總結

從技術定位來看，這個專案展示的是一條以 PyAEDT `0.26.3` 為核心的 Icepak 自動化流程：從 CAD 匯入、材料與熱源配置、網格與求解設定，到最後的溫度結果擷取，全部由 Python 串接完成。它的價值不在於展示複雜演算法，而在於把手機熱模擬中重複、容易出錯的操作整理成可重現的工程流程。
